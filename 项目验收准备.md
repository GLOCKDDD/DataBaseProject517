# 酒店管理系统验收准备文档

## 1. 项目概述

### 1.1 项目名称
酒店管理系统

### 1.2 项目定位
这是一个面向酒店管理员和前台操作员的管理系统，核心目标是把酒店日常业务流程信息化，提升前台业务处理效率，减少人工登记和状态管理中的错误。

### 1.3 项目解决的问题
本系统主要解决以下几个问题：

- 客户信息分散，难以统一管理
- 客房状态不清晰，容易出现重复分配
- 预订和入住之间缺乏联动，可能产生超售问题
- 换房、退房、结账等流程依赖人工处理，容易遗漏
- 会员积分和折扣规则缺乏系统化支持

### 1.4 系统核心业务
系统围绕酒店前台业务形成一个完整闭环：

`客户管理 -> 预订管理 -> 入住登记 -> 换房处理 -> 结账退房 -> 会员积分累计`

### 1.5 项目技术栈

#### 后端
- Java 8
- Spring Boot 2.7.18
- MyBatis-Plus 3.5.3.1
- MySQL 8.0+
- Lombok
- Spring Security Crypto（BCrypt）
- FastJSON

#### 前端
- Vue 3
- Vite
- Vue Router
- Pinia
- Element Plus

## 2. 系统整体架构

### 2.1 架构说明
本项目采用前后端分离的设计思路。

- 前端负责页面展示、用户交互和基础业务流程组织
- 后端负责接口提供、业务规则处理和数据库访问
- 数据库负责业务数据持久化

整体可以概括为：

`前端 Vue -> 后端 Spring Boot -> MySQL 数据库`

### 2.2 项目目录结构

- `hotel-frontend`：前端项目
- `hotel-management-system`：后端项目
- 根目录文档：需求分析、数据库设计说明、README

### 2.3 需要主动说明的一点
当前项目中同时存在：

- 一个基于 Vue 的前端原型版本
- 一套基于 Spring Boot + MySQL 的正式后端实现

前端实现说明文档中提到，前端早期阶段使用 `localStorage` 模拟数据持久化，目的是先把页面和业务流程跑通；后端部分则实现了正式的数据库表结构和业务接口。

答辩时可以这样说：

> 我采用了分阶段开发思路，先用前端原型验证页面流程和业务逻辑，再用 Spring Boot 和 MySQL 实现正式后端与数据库设计，这样可以先保证业务流程可演示，再逐步落地到真实数据库系统中。

## 3. 后端部分介绍

### 3.1 后端技术选型原因
后端选择 Spring Boot，原因是：

- 开发效率高，适合课程设计项目快速搭建
- 分层结构清晰，便于组织控制层、业务层和数据访问层
- 和 MyBatis-Plus 结合后，数据库 CRUD 实现较高效
- 便于后续扩展接口和业务规则

### 3.2 后端架构分层
后端采用典型三层结构：

- `Controller`：接收前端请求并返回结果
- `Service`：处理核心业务逻辑
- `Mapper`：访问数据库

整体流程为：

`Controller -> Service -> Mapper -> MySQL`

### 3.3 后端模块划分
从控制器可以看出，后端主要分为以下模块：

- 用户管理 `UserController`
- 客户管理 `CustomerController`
- 房型管理 `RoomTypeController`
- 房间管理 `RoomController`
- 预订管理 `ReservationController`
- 入住管理 `CheckinController`
- 换房管理 `RoomChangeController`
- 结账管理 `BillController`

### 3.4 后端实现的核心业务规则

#### 1. 用户登录与权限管理
用户登录时，系统会：

- 根据用户名查询用户
- 使用 BCrypt 校验密码
- 更新最后登录时间
- 返回不包含密码的用户信息

系统区分两类角色：

- `admin`：管理员
- `frontdesk`：前台操作员

#### 2. 预订防超售
创建预订时，系统不会直接保存，而是先检查指定时间范围内该房型是否还有可预订数量。

计算逻辑是：

`可预订数 = 房型总房间数 - 当前占用房间数 - 时间重叠的已确认预订数`

只有当可预订数量满足需求时，才允许创建预订。

这体现了项目不仅是简单 CRUD，还包含业务规则控制。

#### 3. 入住登记校验
办理入住时，系统会校验：

- 房间是否存在
- 房间状态是否为空闲
- 如果关联预订，预订状态是否合法
- 入住人数是否超过房型最大容量

入住成功后，会联动更新：

- 新增入住记录
- 新增入住宾客明细
- 房间状态改为“占用”
- 如果关联预订，则预订状态改为“已入住”

#### 4. 换房业务处理
换房时会校验：

- 当前入住记录是否存在
- 入住状态是否允许换房
- 原房间是否处于占用状态
- 新房间是否为空闲状态

换房成功后，系统会：

- 保存换房记录
- 更新入住记录中的房间编号
- 原房间状态改为“清洁中”
- 新房间状态改为“占用”

#### 5. 结账退房处理
结账时，系统会：

- 检查入住记录是否存在
- 检查是否已经结账，防止重复生成账单
- 根据入住天数、房型价格、会员等级自动计算费用
- 生成账单记录
- 更新入住状态、房间状态、预订状态
- 给客户累计会员积分

### 3.5 后端亮点总结
答辩时可以强调：

- 使用分层架构，结构清晰
- 使用 BCrypt 处理密码，不是明文保存
- 使用事务处理结账等多表联动更新
- 业务中实现了防超售、状态联动、重复结账校验等规则

## 4. 数据库设计介绍

### 4.1 数据库选型原因
数据库选择 MySQL，原因是：

- 关系型数据库适合酒店这种结构化业务数据管理
- 适合多表之间建立主外键关系
- 便于进行数据约束、关联查询和事务控制

### 4.2 数据库整体思路
数据库围绕酒店业务流程进行建模，把系统拆分为基础信息、预订、入住、结账和换房几个部分。

### 4.3 核心数据表
项目中主要有以下 10 张核心表：

- `users`：系统用户表
- `customers`：客户信息表
- `room_types`：客房类型表
- `rooms`：客房信息表
- `reservations`：预订主表
- `reservation_details`：预订明细表
- `checkins`：入住登记表
- `checkin_guests`：入住宾客明细表
- `bills`：账单表
- `room_changes`：换房记录表

### 4.4 各表作用说明

#### 1. users
保存系统登录用户，包括管理员和前台操作员。
主要字段：用户名、密码哈希、角色、权限、创建时间、最后登录时间。

#### 2. customers
保存酒店客户信息，包括身份证号、手机号、会员等级和积分。
它既是普通客户表，也承担会员信息管理功能。

#### 3. room_types
保存房型信息，例如单人间、双人间、套房等。
主要字段包括房型名称、基础价格、容纳人数。

#### 4. rooms
保存具体房间信息，例如房号、所属房型、楼层、当前状态。
一个房型可以对应多个具体房间。

#### 5. reservations
保存预订主信息，包括客户、创建人、预计入住时间、预计退房时间、状态等。

#### 6. reservation_details
保存一条预订对应的房型需求明细。
设计成主表和明细表，是为了支持“一笔预订包含多个房型需求”。

#### 7. checkins
保存实际入住记录，记录某次入住分配到了哪一个房间，什么时候入住，什么时候退房。

#### 8. checkin_guests
保存一条入住记录下的宾客明细，用于支持多人入住。

#### 9. bills
保存结账账单，一条入住记录最多对应一条账单。

#### 10. room_changes
保存换房过程，记录原房间、新房间、换房时间和原因。

### 4.5 数据库设计亮点

#### 1. 主外键关系明确
例如：

- `rooms.type_id -> room_types.type_id`
- `reservations.customer_id -> customers.customer_id`
- `checkins.room_id -> rooms.room_id`
- `bills.checkin_id -> checkins.checkin_id`

这样可以保证数据关系清晰。

#### 2. 设计符合业务流程
表结构不是随意拆分，而是严格围绕酒店实际流程设计。
从预订到入住，再到结账和换房，每一步都有对应数据实体。

#### 3. 预订采用主表加明细表
这是数据库设计中的一个规范化体现，避免把多种房型需求硬塞进同一条记录中。

#### 4. 状态字段设计较完整
例如：

- 房间状态：空闲、占用、清洁中、维修中
- 预订状态：已确认、已入住、已取消、已完成
- 入住状态：入住中、已退房、已换房

这些状态字段帮助系统控制业务流转。

### 4.6 数据库部分可以这样总结

> 数据库设计是本项目的核心基础。我把酒店业务拆分成多个实体表，并通过主外键建立联系，使系统能够支持预订、入住、换房、结账等完整流程。同时，状态字段和约束设计也保证了数据的一致性和业务逻辑的可执行性。

## 5. 前端部分介绍

### 5.1 前端技术选型原因
前端使用 Vue 3 + Element Plus，主要原因是：

- Vue 3 上手快，适合课程设计项目开发
- 组件化开发方便拆分页面和复用逻辑
- Element Plus 提供了成熟的表单、表格、弹窗等后台管理组件
- Vite 启动快，开发效率高

### 5.2 前端架构说明
前端采用单页应用结构，主要模块包括：

- `views`：业务页面
- `router`：路由管理
- `stores`：状态管理
- `api/services.js`：业务逻辑服务层
- `api/mock.js`：本地模拟数据层
- `layouts`：后台页面整体布局

### 5.3 路由与权限控制
系统通过 Vue Router 配置页面路由，通过 Pinia 存储登录状态，并在全局路由守卫中控制访问权限。

主要规则是：

- 未登录不能访问业务页面
- 已登录用户不能重复进入登录页
- 不同角色只能访问对应模块

角色访问范围如下：

- `admin`：可以访问全部模块
- `frontdesk`：主要访问客户、预订、入住、换房、结账等前台业务模块

### 5.4 前端页面模块
从 `views` 目录可以看出，前端包含这些主要页面：

- 登录页
- 控制台
- 用户管理
- 客户管理
- 预订管理
- 入住管理
- 换房管理
- 结账管理
- 房型管理
- 房间管理
- 会员管理

### 5.5 前端业务特点

#### 1. 原型化能力较强
前端通过页面、弹窗、表单和表格，把整个酒店业务流程串联了起来，适合用于演示系统业务流程。

#### 2. 封装了业务服务层
在 `services.js` 中，不只是简单保存数据，还模拟了业务校验，比如：

- 预订时间校验
- 房型可预订数量校验
- 入住人数容量校验
- 重复结账防护
- 状态流转控制

#### 3. 页面结构清晰
后台系统常见页面结构比较完整，包括：

- 列表展示
- 查询筛选
- 新增/编辑弹窗
- 删除确认
- 操作后刷新数据

### 5.6 关于前端 localStorage 的说明
前端说明文档中明确提到，项目原型阶段使用 `localStorage` 模拟数据库。

你答辩时可以这样解释：

> 为了先把业务流程和页面交互跑通，我在前端阶段先做了一个本地模拟版本，把业务逻辑封装在服务层，用 localStorage 充当临时数据源。这样可以快速验证预订、入住、结账等流程。后续正式版本再由 Spring Boot + MySQL 承接真实接口和持久化。

这样老师如果问“为什么前端里有 mock 数据”，你就能比较自然地回答。

## 6. 开发过程介绍

### 6.1 开发思路
本项目的开发过程可以概括为四步：

#### 第一步：需求分析
先分析酒店管理系统的核心场景，明确系统至少要覆盖：

- 客户信息管理
- 房型和房间管理
- 预订管理
- 入住登记
- 换房处理
- 结账退房
- 会员积分管理

#### 第二步：数据库设计
根据业务流程抽象实体，完成 E-R 思路到关系表结构的转换，设计主键、外键和状态字段。

#### 第三步：前端原型开发
先用 Vue 3 + Element Plus 搭建页面，把业务流程、角色权限、基础交互和页面结构跑通。

#### 第四步：后端与数据库实现
再基于 Spring Boot 和 MySQL 实现正式的接口、业务规则和数据库持久化，尤其完善登录、预订、入住、结账等核心逻辑。

### 6.2 我在开发中重点考虑的问题

- 如何保证房间不会超售
- 如何让房态随着入住、换房、退房自动变化
- 如何把一个完整的酒店业务流程拆成合理的数据表
- 如何让系统既能演示，又能体现数据库课程设计的特点

### 6.3 项目难点
答辩时可以说难点主要有三个：

#### 1. 预订防超售逻辑
不是简单存一条预订记录，而是需要结合房型总量、当前占用和时间重叠预订来判断可预订数。

#### 2. 多表状态联动
入住、换房、结账都会影响多张表，比如房间表、入住表、预订表、账单表。

#### 3. 业务流程建模
要把酒店业务从现实流程转化为数据库结构和程序逻辑，这比普通的增删改查更复杂。

## 9. 老师可能会问的问题与参考回答

### 9.1 项目总体类问题

#### 问题 1：你这个项目主要是做什么的？
**回答：**
这是一个酒店管理系统，主要服务于酒店管理员和前台操作员，解决客户信息管理、房态管理、预订安排、入住登记、换房处理和结账退房等问题，让酒店日常业务流程更加规范和高效。

#### 问题 2：你这个系统的核心业务流程是什么？
**回答：**
核心流程是客户管理、预订管理、入住登记、换房处理、结账退房以及会员积分累计。也就是从客户产生住宿需求，到完成入住，再到最终结账离店，形成一个完整业务闭环。

#### 问题 3：你的系统有哪些角色？
**回答：**
主要有两个角色，一个是管理员 `admin`，负责用户、房型、房间、会员等后台管理；另一个是前台操作员 `frontdesk`，主要负责客户登记、预订、入住、换房和结账等前台业务。

---

### 9.2 架构与技术类问题

#### 问题 4：为什么选择 Spring Boot + Vue 3？
**回答：**
因为这套技术栈比较适合课程设计项目。Spring Boot 后端开发效率高，分层结构清晰，适合做业务逻辑和数据库接口；Vue 3 组件化开发方便，配合 Element Plus 可以快速搭建后台管理页面。

#### 问题 5：为什么采用前后端分离？
**回答：**
前后端分离可以让页面展示和后端业务逻辑解耦，结构更清晰，也便于后续扩展和维护。前端只负责交互和展示，后端专注处理业务规则和数据库访问。

#### 问题 6：你前端为什么还有 mock 和 localStorage？
**回答：**
这是因为我采用了分阶段开发方式。前期先用前端原型和本地模拟数据把业务流程跑通，验证页面和交互是否合理；后续再用 Spring Boot 和 MySQL 实现正式的后端和数据库。这样做有利于先验证流程，再落地正式实现。

---

### 9.3 数据库设计类问题

#### 问题 7：你数据库里为什么要分这么多表？
**回答：**
因为酒店业务本身就包含多个阶段和对象，比如客户、房型、房间、预订、入住、账单、换房，它们职责不同，拆表后更符合业务逻辑，也更方便建立主外键关系和进行数据约束。

#### 问题 8：为什么预订要设计成主表和明细表？
**回答：**
因为一笔预订可能不只是预订一种房型，可能会同时需要多个房型和数量。如果全部放在一张表里会导致结构不规范，所以我采用预订主表保存整体信息，明细表保存房型需求，这样更符合数据库规范化设计思想。

#### 问题 9：数据库课程设计的体现在哪里？
**回答：**
主要体现在三个方面：第一是实体抽象和表结构设计；第二是主外键约束和多表关系设计；第三是把现实业务规则，比如预订防超售、房态联动、结账更新等，转化为系统中的数据结构和逻辑约束。

#### 问题 10：你的表之间有哪些典型关系？
**回答：**
例如房间表通过 `type_id` 关联房型表，预订表通过 `customer_id` 关联客户表，入住表通过 `room_id` 关联房间表，账单表通过 `checkin_id` 关联入住表。这些关系保证了业务数据之间的关联性。

### 9.4 业务逻辑类问题

#### 问题 11：怎么防止预订超售？
**回答：**
系统会在创建预订前计算某个房型在指定时间段内的可预订数量。计算时会考虑总房间数、当前已占用房间数、以及时间重叠的已确认预订数。如果需求数量超过可预订数量，就不允许创建预订。

#### 问题 12：入住时做了哪些校验？
**回答：**
入住时会校验房间是否存在、房间是否为空闲、如果关联预订则检查预订状态是否合法，同时还会校验入住人数不能超过房型可容纳人数。只有这些条件满足，才允许办理入住。

#### 问题 13：换房时为什么要改旧房间和新房间状态？
**回答：**
因为换房不仅是入住记录变更，还会直接影响房态。旧房间原本有人使用，换出后需要进入清洁流程，所以改成“清洁中”；新房间分配给当前客户后，就应该改成“占用”。

#### 问题 14：结账时为什么要更新多张表？
**回答：**
因为结账不是单一动作，而是业务流程的结束节点。结账完成后，入住记录要改成“已退房”，房间要改成“清洁中”，如果是由预订转入住，还要把预订状态更新为“已完成”，同时生成账单并累积会员积分。

#### 问题 15：你是怎么计算费用的？
**回答：**
费用主要按照入住天数、房型基础价格和会员折扣计算。也就是先算基础房费，再根据会员等级应用折扣，最后生成账单。这个设计能体现会员体系和自动结算逻辑。

### 9.5 安全性与规范性类问题

#### 问题 16：密码是怎么保存的？
**回答：**
正式后端版本里，密码不是明文保存，而是使用 BCrypt 加密后保存到数据库中。登录时不是直接比较字符串，而是用 BCrypt 去匹配输入密码和数据库中的哈希值。

#### 问题 17：为什么结账使用事务？
**回答：**
因为结账会同时修改账单、入住、房间、预订、客户积分等多张表。如果其中一步失败，会导致数据不一致，所以使用事务来保证这些操作要么全部成功，要么全部回滚。

#### 问题 18：你这个项目有什么地方不是简单 CRUD？
**回答：**
主要体现在业务规则部分，比如预订防超售、房态联动、入住人数限制、重复结账防护、会员积分累计等。这些地方都体现了业务逻辑，不只是增删改查。

### 9.6 项目总结类问题

#### 问题 19：你觉得这个项目的难点是什么？
**回答：**
我觉得难点主要有三个：第一是把现实中的酒店业务流程抽象成数据库结构；第二是处理预订、入住、换房、结账过程中的状态联动；第三是保证多表更新时的数据一致性。

#### 问题 20：如果继续完善，你会怎么做？
**回答：**
如果继续完善，我会重点做三方面优化：第一是完成前后端真实联调，统一数据来源；第二是补充更完整的权限控制和操作日志；第三是加入统计报表，比如入住率、营收统计和会员消费分析，让系统更接近实际应用场景。

## 10. 最后建议

1. 先说项目是做什么的
2. 再说系统角色和核心模块

`角色`
`模块`
`表结构`
`业务流程`
`业务规则`

如果老师追问代码细节，你就尽量把回答落到“为什么这样设计、这样设计解决了什么问题”上，不要只说语法和框架名字。
