# 酒店管理系统 - 前端实现说明文档

## 一、实现环境说明

### 1.1 技术栈概览

本项目是一个基于 Vue 3 的单页应用（SPA），采用前后端分离架构设计。由于课程设计阶段尚未对接真实后端服务，所有数据操作通过 `localStorage` 模拟数据库持久化，业务逻辑封装在前端服务层中。

| 类别 | 技术 | 版本 |
|------|------|------|
| 核心框架 | Vue 3（Composition API + `<script setup>`） | ^3.4.0 |
| 路由管理 | Vue Router | ^4.3.0 |
| 状态管理 | Pinia | ^2.1.7 |
| UI 组件库 | Element Plus（中文 locale） | ^2.7.0 |
| 图标库 | @element-plus/icons-vue | ^2.3.1 |
| 构建工具 | Vite | ^5.4.0 |
| 开发语言 | JavaScript（ES Modules） | — |

## 二、登录界面设计

### 2.1 页面布局

登录页采用居中卡片式布局，深色背景（与侧边栏同色 `#0A1628`）配合白色登录卡片，视觉层次分明。卡片顶部有渐变装饰条（蓝→金→蓝），底部展示测试账号信息。

页面包含两个输入字段（用户名、密码）和一个登录按钮，支持回车键触发登录。密码输入框使用 `show-password` 属性提供明文/密文切换功能。

### 2.2 表单校验

使用 Element Plus 的 `el-form` 内置校验机制，对用户名和密码均只做"非空"校验：

```js
// src/views/auth/Login.vue
const form = reactive({ username: '', password: '' })

const rules = {
  username: [{ required: true, message: '请输入用户名', trigger: 'blur' }],
  password: [{ required: true, message: '请输入密码', trigger: 'blur' }]
}
```

### 2.3 登录流程

登录提交时，先通过 `formRef.value.validate()` 触发表单校验，校验通过后调用 Pinia 认证存储的 `login` 方法。为模拟网络请求体验，添加了 500ms 的延迟。登录成功后跳转至 `/dashboard`，失败则弹出错误提示。

```js
// src/views/auth/Login.vue
async function handleLogin() {
  const valid = await formRef.value?.validate().catch(() => false)
  if (!valid) return
  loading.value = true
  setTimeout(() => {
    const result = authStore.login(form.username, form.password)
    if (result.success) {
      ElMessage.success(`欢迎回来，${result.user.username}！`)
      router.push('/dashboard')
    } else {
      ElMessage.error(result.message)
    }
    loading.value = false
  }, 500)
}
```

### 2.4 路由守卫

全局路由守卫实现了三层访问控制：未登录用户重定向到登录页、已登录用户不能重复访问登录页、角色权限校验（`meta.roles`）。

```js
// src/router/index.js
router.beforeEach((to, from, next) => {
  document.title = to.meta.title ? `${to.meta.title} - 酒店管理系统` : '酒店管理系统'
  const authStore = useAuthStore()

  // 未登录 → 重定向登录页
  if (to.meta.requiresAuth !== false && !authStore.isLoggedIn) {
    next('/login'); return
  }
  // 已登录访问登录页 → 跳转首页
  if (to.path === '/login' && authStore.isLoggedIn) {
    next('/dashboard'); return
  }
  // 角色权限校验
  if (to.meta.roles && !to.meta.roles.includes(authStore.currentUser?.role)) {
    next('/dashboard'); return
  }
  next()
})
```

### 2.5 用户角色与权限

系统定义了两种角色，通过路由 `meta.roles` 控制菜单可见性和页面访问：

| 角色 | 标识 | 可访问模块 |
|------|------|-----------|
| 管理员 | `admin` | 全部模块（含用户管理、客房类型、客房信息、会员管理） |
| 前台操作员 | `frontdesk` | 控制台、客户管理、预订、入住、换房、结账 |

权限判断逻辑在 Pinia store 中实现：

```js
// src/stores/auth.js
function hasPermission(perm) {
  if (!currentUser.value) return false
  if (currentUser.value.role === 'admin') return true  // 管理员拥有全部权限
  const perms = currentUser.value.permissions
  if (!perms) return true
  try {
    const permList = typeof perms === 'string' ? JSON.parse(perms) : perms
    return permList.includes(perm)
  } catch { return true }
}
```

## 三、密码校验设计

### 3.1 校验策略概述

本系统的密码校验分为两个层面：**登录时的密码匹配验证** 和 **表单输入的非空校验**。当前版本未实现密码强度校验（如长度、复杂度要求），密码以明文形式存储和比对。

### 3.2 登录密码匹配

认证存储中，`login` 函数从 Mock 数据库获取用户列表，通过 `username` 和 `password_hash` 字段进行精确匹配。匹配成功后，从用户信息中删除密码字段再存入 `sessionStorage`，避免前端泄露密码。

```js
// src/stores/auth.js
function login(username, password) {
  const users = db.getTable('users')
  const user = users.find(u => u.username === username && u.password_hash === password)
  if (!user) {
    return { success: false, message: '用户名或密码错误' }
  }
  user.last_login = new Date().toISOString()
  db.updateRecord('users', user.user_id, { last_login: user.last_login })

  const userInfo = { ...user }
  delete userInfo.password_hash  // 移除密码字段
  currentUser.value = userInfo
  sessionStorage.setItem('currentUser', JSON.stringify(userInfo))
  return { success: true, user: userInfo }
}
```

### 3.3 密码存储方式

Mock 数据层中，密码以明文存储在 `password_hash` 字段中（字段命名为 `password_hash` 是为预留后续加密扩展，当前实际存储明文）。

```js
// src/api/mock.js - 初始用户数据
users: [
  { user_id: 1, username: 'admin',  password_hash: 'admin123', role: 'admin',     ... },
  { user_id: 2, username: 'front1', password_hash: '123456',  role: 'frontdesk', ... }
]
```

### 3.4 表单校验规则

系统在 `validators.js` 中定义了多种业务表单校验器（手机号、身份证号、正整数、正数、时间逻辑等），但未包含密码强度校验器。用户管理页面的创建用户表单同样只做非空校验。


### 3.5 会话管理

用户登录状态存储在 `sessionStorage` 中（key 为 `currentUser`），关闭浏览器标签页后自动清除，比 `localStorage` 更安全。登出时同步清除 Pinia 状态和 sessionStorage。

```js
// src/stores/auth.js
const currentUser = ref(JSON.parse(sessionStorage.getItem('currentUser') || 'null'))
const isLoggedIn = computed(() => !!currentUser.value)

function logout() {
  currentUser.value = null
  sessionStorage.removeItem('currentUser')
}
```

## 四、数据管理设计

### 4.1 整体架构

数据管理采用三层架构：

```
视图层 (Views)  →  服务层 (Services)  →  数据层 (Mock DB / localStorage)
```

- **视图层**：Vue 组件，负责 UI 渲染和用户交互，通过 `onMounted` 挂载时加载数据
- **服务层**：封装所有业务逻辑和校验规则，每个方法为 `async`，模拟 200ms 网络延迟
- **数据层**：封装 `localStorage` 的 CRUD 操作，模拟数据库持久化

### 4.2 Mock 数据层

数据层使用 `localStorage` 的 `hotel_db` key 存储全部数据，提供统一的增删改查接口。新增记录时自动计算自增 ID。

```js
// src/api/mock.js
const STORAGE_KEY = 'hotel_db'

export const db = {
  getTable(tableName) {
    const data = getDB()
    return data[tableName] || []
  },

  addRecord(tableName, record, idField) {
    const data = getDB()
    const table = data[tableName] || []
    const maxId = table.reduce((max, item) => Math.max(max, item[idField] || 0), 0)
    record[idField] = maxId + 1
    table.push(record)
    data[tableName] = table
    saveDB(data)
    return record
  },

  updateRecord(tableName, id, updates, idField) {
    const data = getDB()
    const table = data[tableName] || []
    const idx = table.findIndex(item => item[idField] === id)
    if (idx !== -1) {
      table[idx] = { ...table[idx], ...updates }
      data[tableName] = table
      saveDB(data)
      return table[idx]
    }
    return null
  },

  deleteRecord(tableName, id, idField) {
    const data = getDB()
    const table = data[tableName] || []
    data[tableName] = table.filter(item => item[idField] !== id)
    saveDB(data)
  }
}
```

### 4.4 服务层业务逻辑

服务层是系统的核心，封装了所有业务规则校验。每个服务方法均为异步函数，先执行校验逻辑，通过后再操作数据层。

#### 4.4.1 预订服务 - 超售防护

创建预订时，系统会计算指定时间段内各房型的可预订数量（总房间数 - 已占用 - 已确认且时间重叠的预订），防止超售。

```js
// src/api/services.js
async create(reservation) {
  await delay()
  const { details, ...resData } = reservation

  if (new Date(resData.expected_checkout) <= new Date(resData.expected_checkin)) {
    throw new Error('预计离开时间必须晚于预计入住时间')
  }

  // 超售校验
  for (const detail of details) {
    const available = await roomService.getAvailableCount(
      detail.type_id, resData.expected_checkin, resData.expected_checkout
    )
    if (detail.room_count > available) {
      const typeName = db.getTable('room_types').find(t => t.type_id === detail.type_id)?.type_name
      throw new Error(`${typeName}可预订数量不足，当前可预订${available}间`)
    }
  }
  // ... 保存预订和明细
}
```

可预订数量的计算逻辑：

```js
// src/api/services.js
async getAvailableCount(typeId, checkinDate, checkoutDate) {
  const rooms = db.getTable('rooms').filter(r => r.type_id === typeId)
  const totalRooms = rooms.length
  const occupiedCount = rooms.filter(r => r.status === '占用').length

  // 时间段重叠判断
  const overlappingReservations = reservations.filter(r => {
    if (r.status !== '已确认') return false
    return new Date(r.expected_checkin) < new Date(checkoutDate) &&
           new Date(r.expected_checkout) > new Date(checkinDate)
  })
  const reservedCount = overlappingReservations.reduce(/* 汇总间数 */)
  return totalRooms - occupiedCount - reservedCount
}
```

#### 4.4.2 入住服务 - 多重校验

入住办理时依次校验：房间状态必须为"空闲"、关联预订状态合法、入住人数不超过房间容量。

```js
// src/api/services.js
async create(checkinData) {
  await delay()
  const { guests, reservation_id, room_id } = checkinData

  const room = db.findById('rooms', room_id, 'room_id')
  if (room.status !== '空闲') throw new Error('该房间当前不可分配')

  if (reservation_id) {
    const reservation = db.findById('reservations', reservation_id, 'reservation_id')
    if (!['已确认', '已入住'].includes(reservation.status)) {
      throw new Error('预订状态不合法')
    }
  }

  const roomType = db.findById('room_types', room.type_id, 'type_id')
  if (guests.length > roomType.capacity) {
    throw new Error(`入住人数超过房间最大容量(${roomType.capacity})`)
  }
  // ... 创建入住记录、更新房间状态为"占用"
}
```

#### 4.4.3 结账服务 - 费用计算与积分累计

结账时计算房费（天数 × 基础价 × 会员折扣 + 额外费用），生成账单后联动更新多个表的状态，并为客户累计积分。

```js
// src/api/services.js
async checkout(checkinId, extraCharges = []) {
  await delay()
  // ... 校验入住状态、防重复结账

  const nights = daysBetween(checkinTime, checkoutTime)
  const discount = customer ? (MEMBER_DISCOUNTS[customer.membership_level] || 1.0) : 1.0
  const roomFee = nights * roomType.base_price * discount
  const totalAmount = roomFee + extraCharges.reduce((sum, c) => sum + c.amount, 0)

  // 联动更新：入住→已退房、房间→清洁中、预订→已完成
  db.updateRecord('checkins', checkinId, { status: '已退房', checkout_time: new Date().toISOString() })
  db.updateRecord('rooms', checkin.room_id, { status: '清洁中' })
  if (checkin.reservation_id) {
    db.updateRecord('reservations', checkin.reservation_id, { status: '已完成' })
  }
  // 累计积分（1元=1积分），自动检查会员升级
  await customerService.addPoints(customer.customer_id, Math.floor(bill.total_amount))
}
```

### 4.5 状态机设计

系统通过枚举和状态流转校验函数，确保业务状态只能按合法路径转换。

```js
// src/utils/helpers.js
// 房间状态流转：空闲→占用/维修中→清洁中→空闲
export function isValidRoomTransition(from, to) {
  const transitions = {
    '空闲': ['占用', '维修中'],
    '占用': ['清洁中'],
    '清洁中': ['空闲'],
    '维修中': ['清洁中']
  }
  return transitions[from]?.includes(to) || false
}

// 预订状态流转：已确认→已入住/已取消→已完成
export function isValidReservationTransition(from, to) {
  const transitions = {
    '已确认': ['已入住', '已取消'],
    '已入住': ['已完成']
  }
  return transitions[from]?.includes(to) || false
}
```

### 4.6 会员积分系统

会员分为三个等级，不同等级享受不同折扣。结账时 1 元消费累计 1 积分，积分达到阈值自动升级。

```js
// src/utils/helpers.js
export const MEMBER_DISCOUNTS = { '普通': 1.0, 'VIP': 0.9, '贵宾': 0.8 }
export const MEMBER_UPGRADE_THRESHOLDS = { '普通': 0, 'VIP': 1000, '贵宾': 5000 }
```

自动升级检查逻辑：

```js
// src/api/services.js
async addPoints(customerId, points) {
  const customer = db.findById('customers', customerId, 'customer_id')
  customer.points += points
  const currentLevelIdx = MEMBER_LEVEL_LIST.findIndex(l => l.value === customer.membership_level)
  for (let i = MEMBER_LEVEL_LIST.length - 1; i > currentLevelIdx; i--) {
    if (customer.points >= MEMBER_UPGRADE_THRESHOLDS[MEMBER_LEVEL_LIST[i].value]) {
      customer.membership_level = MEMBER_LEVEL_LIST[i].value
      break
    }
  }
  return db.updateRecord('customers', customerId, customer, 'customer_id')
}
```
